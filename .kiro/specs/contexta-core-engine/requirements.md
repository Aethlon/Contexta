# Requirements Document

## Introduction

contexta Core Engine is the central intelligence pipeline that powers the contexta Memory Intelligence Engine for AI agents. It encompasses the full lifecycle of memory management: observing agent conversations, extracting structured memories, classifying and scoring them, resolving contradictions with existing knowledge, storing memories in a hybrid index, and retrieving relevant context on demand. Beyond basic memory CRUD, the core engine includes an importance framework with type-based scoring rules, a policy engine for domain-specific extraction behavior, a schema registry for structured memory types, a decay engine for memory lifecycle transitions, a reflection engine for autonomous memory maintenance, a retrieval feedback loop for measured utility scoring, and an entity state manager for rich entity models. The core engine is the foundational layer that all developer-facing SDKs and surfaces depend on.

## Glossary

- **Observation_Engine**: The subsystem that receives raw conversation data (user messages, assistant messages, tool outputs, metadata, full sessions) and produces structured observations
- **Memory_Record**: A single unit of stored intelligence containing metadata, content, scores, embeddings, and lifecycle information
- **Extraction_Worker**: The Python-based service that processes observations to extract typed memories (facts, preferences, goals, projects, skills, relationships, events, patterns, contacts)
- **Entity_Resolver**: The subsystem that links observations across sessions to unified entities and maintains a graph of relationships
- **Truth_Maintenance_Engine**: The subsystem that detects contradictions between new and existing memories, invalidates outdated truths, and creates supersession links
- **Memory_Scoring_Engine**: The subsystem that computes importance, confidence, freshness, and utility scores for each memory
- **Retrieval_Engine**: The subsystem that combines semantic search, keyword search, graph expansion, recency ranking, and importance ranking to find relevant memories
- **Context_Builder**: The subsystem that assembles retrieved memories into structured context suitable for agent consumption
- **Memory_Graph**: The entity-relationship graph connecting Projects, Persons, Companies, Technologies, Preferences, Goals, Skills, and Topics via typed edges
- **Sensitive_Data**: Passwords, OTPs, payment card numbers, API secrets, tokens, and session cookies that must never be stored
- **Supersession**: The process of invalidating an old memory and linking it to a new memory via a SUPERSEDED_BY relationship
- **Session**: A bounded conversation between a user and an agent, identified by a session_id
- **Tenant**: An organization using contexta, identified by organization_id, providing data isolation boundaries
- **Hot_Context**: Frequently accessed or recently relevant memories cached in Redis for fast retrieval
- **Epilogue_Worker**: The asynchronous Python worker that performs extraction, classification, contradiction detection, and memory upsert after observation
- **Importance_Framework**: The rule-based system that assigns base importance scores by memory_type and applies modifiers based on repetition, recency, user emphasis, and decision impact
- **Policy_Engine**: The subsystem that allows developers to configure domain-specific extraction and storage rules (what to save, what to ignore) via named policy profiles
- **Memory_Policy**: A named configuration profile that defines extraction rules, memory type priorities, and ignore patterns for a specific agent domain (e.g., coding-agent, crm-agent, tutor-agent)
- **Schema_Registry**: The subsystem that allows developers to register custom structured memory schemas, enabling typed extraction into developer-defined fields
- **Custom_Schema**: A developer-defined data structure registered with the Schema_Registry that specifies named fields, types, and validation rules for a memory category
- **Decay_Engine**: The subsystem that manages memory lifecycle state transitions (active → warm → cold → archived) based on age, access patterns, and relevance signals
- **Memory_State**: The lifecycle state of a memory: active (frequently relevant), warm (occasionally relevant), cold (rarely accessed), or archived (excluded from retrieval)
- **Reflection_Engine**: The autonomous background worker that performs periodic memory maintenance including duplicate merging, contradiction detection, dormant goal identification, project snapshot generation, and user model synthesis
- **User_Model**: A synthesized profile generated by the Reflection_Engine containing aggregated user attributes (languages, projects, preferences, active goals, skills) used for context enrichment
- **Project_Snapshot**: A periodic summary of a project entity generated by the Reflection_Engine containing technologies used, status, open tasks, and key relationships
- **Retrieval_Feedback_Engine**: The subsystem that tracks which retrieved memories were actually used by agents, enabling measured (not guessed) utility scoring
- **Entity_State_Manager**: The subsystem that maintains rich state for each entity in the Memory_Graph including summary, status, last_updated, and aggregated attributes
- **Memory_Compression_Engine**: The subsystem that generates compressed summaries from clusters of related memories, reducing context size while preserving intelligence
- **Compressed_Summary**: A generated summary that distills multiple related memories about an entity or topic into a single concise representation for efficient context delivery
- **Semantic_Cluster_Engine**: The subsystem that automatically groups related memories into semantic clusters (e.g., "Backend Infrastructure" grouping Postgres, Redis, Go, Docker) for cluster-based retrieval
- **Semantic_Cluster**: A named group of semantically related memories and entities that can be retrieved as a unit
- **Dream_Cycle_Engine**: The autonomous evaluation subsystem that generates synthetic questions, evaluates retrieval quality, identifies missing knowledge gaps, and drives memory self-improvement
- **Missing_Memory_Candidate**: A knowledge gap identified by the Dream_Cycle_Engine where related information exists but a specific field or fact is unknown
- **Context_Planner**: The subsystem that allocates token budget across memory categories (projects, goals, preferences, events, facts) to optimize context delivery within agent context window limits
- **Token_Budget**: A configurable allocation of context window capacity across memory categories, expressed as percentages or absolute token counts
- **Episodic_Memory**: A first-class memory type representing time-bound events, experiences, failures, and milestones (distinct from facts, which are timeless truths)

## Requirements

### Requirement 1: Observation Ingestion

**User Story:** As a developer, I want to send full conversation context to contexta, so that the system can extract and store useful long-term intelligence from agent interactions.

#### Acceptance Criteria

1. WHEN a conversation payload containing user messages, assistant messages, tool outputs, and metadata is submitted, THE Observation_Engine SHALL accept and validate the payload within 500ms
2. WHEN a valid observation payload is received, THE Observation_Engine SHALL enqueue the payload for asynchronous extraction processing
3. IF an observation payload exceeds the maximum allowed size of 1MB, THEN THE Observation_Engine SHALL reject the payload with a descriptive error indicating the size limit
4. IF an observation payload contains malformed or missing required fields, THEN THE Observation_Engine SHALL reject the payload with a validation error specifying the invalid fields
5. WHEN an observation payload is accepted, THE Observation_Engine SHALL associate the payload with the correct tenant (organization_id) and user (user_id)
6. THE Observation_Engine SHALL support batch submission of multiple session observations in a single request

### Requirement 2: Memory Extraction and Classification

**User Story:** As a developer, I want contexta to automatically extract structured memories from conversations, so that useful intelligence is captured without manual effort.

#### Acceptance Criteria

1. WHEN an observation payload is dequeued for processing, THE Extraction_Worker SHALL extract structured memories of types: fact, preference, goal, project, skill, relationship, event, episodic, pattern, contact, and custom
2. WHEN a memory is extracted, THE Extraction_Worker SHALL assign a memory_type classification from the defined set of types
3. WHEN a memory is extracted, THE Extraction_Worker SHALL assign a source_type classification (user_explicit, agent_inference, tool_output, imported_file, or api)
4. WHEN a memory is extracted, THE Extraction_Worker SHALL generate a title and structured_data representation for the memory
5. WHEN extraction is complete for a payload, THE Extraction_Worker SHALL emit extracted memories for downstream processing (deduplication, contradiction detection, scoring, storage)
6. IF the Extraction_Worker encounters an unprocessable payload, THEN THE Extraction_Worker SHALL log the failure with diagnostic details and mark the extraction job as failed

### Requirement 3: Sensitive Data Filtering

**User Story:** As a developer, I want contexta to automatically filter sensitive data at ingestion time, so that passwords, tokens, and secrets are never stored in the memory system.

#### Acceptance Criteria

1. WHEN an observation payload is received, THE Observation_Engine SHALL scan the payload content for Sensitive_Data patterns before enqueuing for extraction
2. WHEN Sensitive_Data is detected in an observation payload, THE Observation_Engine SHALL redact the sensitive content from the payload before processing
3. THE Observation_Engine SHALL detect the following Sensitive_Data types: passwords, one-time passwords, payment card numbers, API secrets, authentication tokens, and session cookies
4. WHEN a memory is extracted, THE Extraction_Worker SHALL perform a secondary scan to verify no Sensitive_Data remains in the extracted memory content
5. IF Sensitive_Data is detected during secondary scan, THEN THE Extraction_Worker SHALL discard the affected memory and log a security event

### Requirement 4: Entity Resolution

**User Story:** As a developer, I want contexta to link related observations across sessions to unified entities, so that the system builds a coherent knowledge graph over time.

#### Acceptance Criteria

1. WHEN a new memory is extracted, THE Entity_Resolver SHALL attempt to match the memory to existing entities in the Memory_Graph
2. WHEN a match is found with confidence above 0.8, THE Entity_Resolver SHALL link the new memory to the existing entity
3. WHEN no match is found, THE Entity_Resolver SHALL create a new entity node in the Memory_Graph
4. THE Entity_Resolver SHALL support entity types: Project, Person, Company, Technology, Preference, Goal, Skill, and Topic
5. THE Entity_Resolver SHALL create typed edges between entities using relationship types: USES, WORKS_ON, LIKES, DEPENDS_ON, OWNS, SUPERSEDED_BY, and RELATED_TO
6. WHEN an entity is resolved, THE Entity_Resolver SHALL update the entity metadata with the latest observation timestamp

### Requirement 5: Truth Maintenance and Contradiction Resolution

**User Story:** As a developer, I want contexta to automatically detect and resolve contradictions between new and existing memories, so that the system always reflects the most current truth.

#### Acceptance Criteria

1. WHEN a new memory is extracted, THE Truth_Maintenance_Engine SHALL compare the new memory against existing memories for the same entity and memory_type
2. WHEN a contradiction is detected between a new memory and an existing memory, THE Truth_Maintenance_Engine SHALL invalidate the existing memory by setting its valid_to timestamp
3. WHEN an existing memory is invalidated, THE Truth_Maintenance_Engine SHALL create a SUPERSEDED_BY link from the old memory to the new memory
4. WHEN a contradiction is resolved, THE Truth_Maintenance_Engine SHALL preserve the old memory in memory_versions for historical reference
5. THE Truth_Maintenance_Engine SHALL maintain both current truth (valid_to is null) and historical truth (valid_to is set) for each entity
6. WHEN a supersession occurs, THE Truth_Maintenance_Engine SHALL log the contradiction event with both old and new memory content for explainability

### Requirement 6: Memory Scoring

**User Story:** As a developer, I want each memory to receive computed scores for importance, confidence, freshness, and utility, so that retrieval can prioritize the most relevant memories.

#### Acceptance Criteria

1. WHEN a memory is stored, THE Memory_Scoring_Engine SHALL compute an importance score between 0.0 and 1.0 by delegating to the Importance_Framework which applies type-based base scores and contextual modifiers
2. WHEN a memory is stored, THE Memory_Scoring_Engine SHALL assign a confidence score between 0.0 and 1.0 based on source_type (user_explicit: 1.0, tool_output: 0.8, agent_inference: 0.6)
3. THE Memory_Scoring_Engine SHALL compute a freshness score that decays over time based on the memory age relative to the current timestamp
4. THE Memory_Scoring_Engine SHALL compute a utility score that reflects how often the memory has been retrieved and found useful
5. WHEN a memory is retrieved and used in context, THE Memory_Scoring_Engine SHALL increment the utility score for that memory
6. THE Memory_Scoring_Engine SHALL recompute freshness scores on retrieval rather than storing a static value

### Requirement 7: Memory Storage

**User Story:** As a developer, I want memories to be stored in a hybrid index with full metadata, so that they can be efficiently retrieved through multiple access patterns.

#### Acceptance Criteria

1. WHEN a memory passes extraction, deduplication, and contradiction resolution, THE Memory_Record SHALL be persisted to PostgreSQL with all required fields (id, user_id, org_id, memory_type, title, content, structured_data, source_type, confidence, importance, tags, session_id, created_at, updated_at, valid_from)
2. WHEN a memory is stored, THE Embedding_Service SHALL generate a vector embedding and store it in pgvector alongside the Memory_Record
3. THE Memory_Record storage SHALL enforce tenant isolation by requiring organization_id on all records and restricting queries to the authenticated tenant
4. WHEN a memory is stored, THE system SHALL index the memory for keyword search, semantic search, and graph traversal
5. THE system SHALL store graph edges in PostgreSQL representing relationships between entities
6. WHEN a memory is pinned by a user, THE system SHALL set is_pinned to true and exclude the memory from decay and archival processes

### Requirement 8: Hybrid Retrieval

**User Story:** As a developer, I want to retrieve relevant memories using a combination of semantic search, keyword search, graph expansion, and ranking, so that the most contextually appropriate memories are returned.

#### Acceptance Criteria

1. WHEN a retrieval query is submitted, THE Retrieval_Engine SHALL execute semantic search using vector similarity against stored embeddings
2. WHEN a retrieval query is submitted, THE Retrieval_Engine SHALL execute keyword search against memory titles, content, and tags
3. WHEN a retrieval query is submitted, THE Retrieval_Engine SHALL perform graph expansion to find related entities within a configurable hop depth
4. THE Retrieval_Engine SHALL combine results using a weighted formula: 40% semantic similarity, 25% graph relevance, 20% importance score, 10% recency score, 5% keyword match
5. WHEN combined results are computed, THE Retrieval_Engine SHALL apply LLM-based reranking to produce the final ordered result set
6. THE Retrieval_Engine SHALL return results within 200ms for queries against memory stores containing up to 100,000 records per tenant
7. THE Retrieval_Engine SHALL respect tenant isolation by restricting all searches to the authenticated organization_id and user_id

### Requirement 9: Context Building

**User Story:** As a developer, I want contexta to assemble retrieved memories into structured context, so that agents receive well-organized intelligence without seeing raw memory records.

#### Acceptance Criteria

1. WHEN context is requested for a session, THE Context_Builder SHALL assemble a structured response containing: user_profile, active_projects, preferences, goals, recent_events, and relevant_memories
2. THE Context_Builder SHALL accept configuration parameters for: number of recent messages, number of relevant memories, graph context depth, whether to include user model, and total token budget
3. WHEN a token budget is specified, THE Context_Builder SHALL delegate allocation decisions to the Context_Planner
4. WHEN context is assembled, THE Context_Builder SHALL filter out archived and invalidated memories from the result
5. WHEN context is assembled, THE Context_Builder SHALL order memories within each section by combined relevance score
6. THE Context_Builder SHALL cache assembled context in Redis (Hot_Context) for repeated access within the same session
7. WHEN cached context exists and no new memories have been stored since cache creation, THE Context_Builder SHALL return the cached version

### Requirement 10: Memory Deduplication

**User Story:** As a developer, I want contexta to detect and prevent duplicate memories from being stored, so that the memory store remains clean and efficient.

#### Acceptance Criteria

1. WHEN a new memory is extracted, THE Extraction_Worker SHALL compare the memory against existing memories using semantic similarity
2. WHEN semantic similarity between a new memory and an existing memory exceeds 0.95, THE Extraction_Worker SHALL treat the new memory as a duplicate
3. WHEN a duplicate is detected, THE Extraction_Worker SHALL discard the new memory and update the existing memory timestamp
4. WHEN a near-duplicate is detected (similarity between 0.85 and 0.95), THE Extraction_Worker SHALL merge the new information into the existing memory record
5. THE Extraction_Worker SHALL scope deduplication checks to the same user_id, organization_id, and memory_type

### Requirement 11: Memory Explainability

**User Story:** As a developer, I want to understand why a specific memory was stored or retrieved, so that I can debug and trust the memory system.

#### Acceptance Criteria

1. WHEN contexta.explain(memoryId) is called, THE system SHALL return the extraction source (session_id, message content that triggered extraction)
2. WHEN contexta.explain(memoryId) is called, THE system SHALL return the classification reasoning (why this memory_type was assigned)
3. WHEN contexta.explain(memoryId) is called, THE system SHALL return the scoring breakdown (importance, confidence, freshness, utility with explanations)
4. WHEN a memory has been superseded, THE system SHALL include the supersession history (previous values and timestamps of change)
5. WHEN contexta.timeline(userId) is called, THE system SHALL return a chronological sequence of memory creation, update, and supersession events for the specified user

### Requirement 12: Memory Lifecycle Management

**User Story:** As a developer, I want to manage memory lifecycle through pinning, archiving, and deletion, so that users maintain control over their stored intelligence.

#### Acceptance Criteria

1. WHEN a pin request is received for a memory, THE system SHALL set is_pinned to true and exclude the memory from automated decay
2. WHEN an unpin request is received for a memory, THE system SHALL set is_pinned to false and re-enable automated decay for the memory
3. WHEN an archive request is received for a memory, THE system SHALL set is_archived to true and exclude the memory from retrieval results
4. WHEN a restore request is received for an archived memory, THE system SHALL set is_archived to false and re-include the memory in retrieval results
5. WHEN a delete request is received for a memory, THE system SHALL permanently remove the Memory_Record and associated embeddings, graph edges, and version history
6. THE system SHALL enforce that lifecycle operations are scoped to the authenticated user_id and organization_id

### Requirement 13: Session Management

**User Story:** As a developer, I want contexta to track sessions and associate memories with their originating sessions, so that I can trace the provenance of stored intelligence.

#### Acceptance Criteria

1. WHEN a new session is initiated, THE system SHALL create a session record with session_id, user_id, organization_id, and start timestamp
2. WHEN messages are received within a session, THE system SHALL store messages with their session_id association
3. WHEN a session ends, THE system SHALL trigger the Epilogue_Worker to process the full session for memory extraction
4. THE system SHALL cache active session data in Redis for low-latency access during ongoing conversations
5. WHEN contexta.inspect(userId) is called, THE system SHALL return session history with associated memories for the specified user

### Requirement 14: Tenant Isolation and Multi-tenancy

**User Story:** As a developer, I want complete data isolation between organizations, so that one tenant can never access another tenant's memories.

#### Acceptance Criteria

1. THE system SHALL require organization_id on every Memory_Record, session, entity, and relationship
2. THE system SHALL enforce organization_id filtering on all database queries at the data access layer
3. THE system SHALL validate that the authenticated tenant matches the organization_id on every write and read operation
4. IF a request attempts to access data belonging to a different organization_id, THEN THE system SHALL reject the request with an authorization error
5. THE system SHALL use shared database tables with organization_id column for tenant isolation (not per-tenant schemas)

### Requirement 15: Embedding Generation

**User Story:** As a developer, I want memories to have vector embeddings generated automatically, so that semantic search can find contextually relevant memories.

#### Acceptance Criteria

1. WHEN a memory is stored or updated, THE Embedding_Service SHALL generate a vector embedding from the memory title and content
2. THE Embedding_Service SHALL store embeddings in pgvector associated with the Memory_Record id
3. WHEN a memory is updated, THE Embedding_Service SHALL regenerate the embedding to reflect the updated content
4. THE Embedding_Service SHALL process embedding generation asynchronously to avoid blocking the storage pipeline
5. IF the Embedding_Service is unavailable, THEN THE system SHALL store the memory without an embedding and enqueue the memory for embedding generation when the service recovers

### Requirement 16: Audit Logging

**User Story:** As a developer, I want all significant operations to be logged for audit purposes, so that I can track who accessed or modified memories and when.

#### Acceptance Criteria

1. WHEN a memory is created, updated, deleted, pinned, unpinned, archived, or restored, THE system SHALL create an audit log entry with the operation type, actor, timestamp, and affected memory_id
2. WHEN a retrieval query is executed, THE system SHALL log the query parameters, result count, and requesting user
3. WHEN a supersession occurs, THE system SHALL log both the old and new memory identifiers in the audit trail
4. THE system SHALL retain audit logs for a minimum of 90 days
5. THE system SHALL scope audit log access to the authenticated organization_id

### Requirement 17: Memory Importance Framework

**User Story:** As a developer, I want memory importance to be computed using base rules combined with learned signals from usage, repetition, and decision impact, so that scoring adapts over time rather than remaining static.

#### Acceptance Criteria

1. THE Importance_Framework SHALL assign base importance scores by memory_type: project (0.9), goal (0.85), preference (0.75), skill (0.7), relationship (0.7), fact (0.6), episodic (0.55), event (0.5), pattern (0.5), contact (0.6), custom (0.5)
2. WHEN a memory has been mentioned in multiple sessions (3 or more), THE Importance_Framework SHALL apply a repetition modifier that increases the base score by up to 0.1
3. WHEN a memory has been referenced within the last 7 days, THE Importance_Framework SHALL apply a recency modifier that increases the base score by up to 0.05
4. WHEN a user explicitly states "remember this" or equivalent emphasis phrases, THE Importance_Framework SHALL apply an emphasis modifier that increases the base score by 0.15
5. WHEN a memory affects future decisions (references goals, deadlines, or dependencies), THE Importance_Framework SHALL apply a decision-impact modifier that increases the base score by up to 0.1
6. THE Importance_Framework SHALL incorporate the measured utility score from the Retrieval_Feedback_Engine as a usage-based modifier (high usage ratio increases importance by up to 0.1, low usage ratio decreases by up to 0.1)
7. THE Importance_Framework SHALL cap the final importance score at 1.0 and floor at 0.0 after all modifiers are applied
8. WHEN a memory content matches low-value patterns (greetings, small talk, filler), THE Importance_Framework SHALL reject the memory from storage entirely
9. THE Importance_Framework SHALL recompute the final importance score periodically (during Reflection Engine cycles) to incorporate accumulated usage and recency signals

### Requirement 18: Memory Policy Engine

**User Story:** As a developer, I want to configure domain-specific extraction policies for different agent types, so that contexta stores only relevant intelligence for each use case.

#### Acceptance Criteria

1. THE Policy_Engine SHALL allow developers to register named Memory_Policy profiles via the SDK
2. WHEN an observation is submitted with a policy parameter, THE Policy_Engine SHALL apply the specified Memory_Policy to control extraction behavior
3. THE Memory_Policy SHALL define store rules specifying which memory_types and content patterns to extract and persist
4. THE Memory_Policy SHALL define ignore rules specifying which content patterns to discard without extraction
5. THE Memory_Policy SHALL define priority weights that override default importance scores for specific memory_types within the policy context
6. WHEN no policy is specified on an observation, THE Policy_Engine SHALL apply the default policy that extracts all supported memory_types
7. THE Policy_Engine SHALL provide built-in policy templates for common agent domains: coding-agent (stores tech stack, repos, preferences, architecture; ignores small talk), crm-agent (stores budget, contacts, requirements, timeline; ignores jokes), and tutor-agent (stores weak topics, strengths, goals, mistakes; ignores off-topic chat)
8. THE Policy_Engine SHALL allow developers to extend built-in templates with custom store and ignore rules

### Requirement 19: Memory Schema Registry

**User Story:** As a developer, I want to register custom structured schemas for memory extraction, so that memories are stored as typed structured data rather than unstructured text blobs.

#### Acceptance Criteria

1. THE Schema_Registry SHALL allow developers to register Custom_Schema definitions specifying field names, field types, and validation rules
2. WHEN a Custom_Schema is registered, THE Schema_Registry SHALL validate the schema definition for correctness (no duplicate fields, valid types, required fields specified)
3. WHEN extraction occurs and a registered Custom_Schema matches the memory content, THE Extraction_Worker SHALL extract data into the schema-defined structured_data fields
4. WHEN a memory is extracted using a Custom_Schema, THE Extraction_Worker SHALL validate the extracted data against the schema field types and required field constraints
5. IF extracted data fails schema validation, THEN THE Extraction_Worker SHALL store the memory with raw content and flag it for manual review
6. THE Schema_Registry SHALL support field types: string, number, boolean, date, enum, array, and nested object
7. THE Schema_Registry SHALL scope registered schemas to the organization_id of the registering tenant
8. WHEN contexta.explain(memoryId) is called for a schema-typed memory, THE system SHALL include the schema name and field mapping in the explanation

### Requirement 20: Memory Decay Engine

**User Story:** As a developer, I want memories to transition through lifecycle states based on age and access patterns, so that stale memories naturally fade while active memories remain prominent.

#### Acceptance Criteria

1. THE Decay_Engine SHALL manage four Memory_State levels: active, warm, cold, and archived
2. WHEN a memory has not been accessed for 30 days, THE Decay_Engine SHALL transition the memory from active to warm state
3. WHEN a memory has not been accessed for 90 days, THE Decay_Engine SHALL transition the memory from warm to cold state
4. WHEN a memory has not been accessed for 180 days, THE Decay_Engine SHALL transition the memory from cold to archived state
5. WHEN a memory in warm or cold state is accessed during retrieval, THE Decay_Engine SHALL transition the memory back to active state
6. THE Decay_Engine SHALL exclude pinned memories from all automated state transitions
7. WHILE a memory is in cold state, THE Retrieval_Engine SHALL apply a ranking penalty of 0.3 to the memory relevance score
8. WHILE a memory is in archived state, THE Retrieval_Engine SHALL exclude the memory from retrieval results unless explicitly requested
9. THE Decay_Engine SHALL run state transition evaluations as a periodic background job (daily)

### Requirement 21: Reflection Engine

**User Story:** As a developer, I want contexta to autonomously maintain memory quality through periodic reflection cycles, so that the memory store self-organizes and improves over time.

#### Acceptance Criteria

1. THE Reflection_Engine SHALL execute as a periodic background worker (configurable interval, default nightly)
2. WHEN the Reflection_Engine detects 3 or more memories containing the same fact for a user, THE Reflection_Engine SHALL merge duplicate memories into a single consolidated memory
3. WHEN the Reflection_Engine detects contradictory memories that were not caught at ingestion time, THE Reflection_Engine SHALL trigger the Truth_Maintenance_Engine to resolve the contradiction
4. WHEN a goal-type memory has not been referenced in any session for 180 days, THE Reflection_Engine SHALL mark the goal as dormant and reduce its importance score by 0.3
5. THE Reflection_Engine SHALL generate a Project_Snapshot for each active project entity containing: technologies used, current status, open tasks, and key relationships
6. THE Reflection_Engine SHALL generate a User_Model for each active user containing: primary languages, active projects, stated preferences, active goals, and identified skills
7. WHEN a User_Model is generated, THE Context_Builder SHALL include the User_Model in context responses when the withUserModel option is enabled
8. WHEN a Project_Snapshot is generated, THE Context_Builder SHALL include relevant Project_Snapshots in the active_projects section of context responses
9. THE Reflection_Engine SHALL invoke the Memory_Compression_Engine to generate Compressed_Summaries for entities with more than 20 associated memories
10. THE Reflection_Engine SHALL invoke the Semantic_Cluster_Engine to identify and update semantic clusters based on new memory relationships
11. THE Reflection_Engine SHALL log all maintenance actions (merges, dormancy marks, compressions, cluster updates, snapshot generations) in the audit trail

### Requirement 22: Retrieval Feedback Engine

**User Story:** As a developer, I want retrieval quality to be measured based on actual memory usage by agents, so that utility scores reflect real-world value rather than guesses.

#### Acceptance Criteria

1. WHEN memories are retrieved and included in agent context, THE Retrieval_Feedback_Engine SHALL track which memories were retrieved for each context request
2. WHEN an agent uses a retrieved memory (references it in a response or action), THE Retrieval_Feedback_Engine SHALL record a positive usage signal for that memory
3. WHEN a retrieved memory is not referenced by the agent in the subsequent response, THE Retrieval_Feedback_Engine SHALL record an ignored signal for that memory
4. THE Retrieval_Feedback_Engine SHALL compute a measured utility score based on the ratio of positive usage signals to total retrieval count for each memory
5. WHEN a memory accumulates 10 or more ignored signals with zero positive usage signals, THE Retrieval_Feedback_Engine SHALL reduce the memory importance score by 0.1
6. WHEN a memory accumulates a usage ratio above 0.8 over 10 or more retrievals, THE Retrieval_Feedback_Engine SHALL increase the memory importance score by 0.05
7. THE Retrieval_Feedback_Engine SHALL store feedback data in a retrieval_feedback table with fields: memory_id, session_id, retrieved_at, signal (used, ignored), and context_request_id
8. THE Retrieval_Feedback_Engine SHALL scope all feedback data to the authenticated organization_id and user_id

### Requirement 23: Entity State Manager

**User Story:** As a developer, I want entities in the memory graph to maintain rich state including summaries, status, and aggregated attributes, so that entities are first-class objects rather than simple graph nodes.

#### Acceptance Criteria

1. THE Entity_State_Manager SHALL maintain a state record for each entity containing: entity_type, name, summary, status (active, inactive, completed, archived), last_updated, and aggregated_attributes
2. WHEN a new memory is linked to an entity, THE Entity_State_Manager SHALL update the entity summary to reflect the new information
3. WHEN a new memory is linked to an entity, THE Entity_State_Manager SHALL update the entity last_updated timestamp
4. THE Entity_State_Manager SHALL compute aggregated_attributes by collecting all facts, preferences, and relationships associated with the entity into a structured view
5. WHEN an entity has not received new observations for 90 days, THE Entity_State_Manager SHALL transition the entity status to inactive
6. WHEN contexta.inspect(userId) is called, THE system SHALL return entity states with their summaries, statuses, and aggregated attributes
7. THE Entity_State_Manager SHALL support entity status transitions: active → inactive → archived, and inactive → active (when new observations arrive)
8. WHEN the Reflection_Engine generates a Project_Snapshot, THE Entity_State_Manager SHALL update the corresponding project entity state with the snapshot data

### Requirement 24: Memory Compression Engine

**User Story:** As a developer, I want clusters of related memories to be compressed into concise summaries, so that context delivery is efficient and agents receive distilled intelligence rather than hundreds of raw memories.

#### Acceptance Criteria

1. WHEN an entity has more than 20 associated memories, THE Memory_Compression_Engine SHALL generate a Compressed_Summary that distills the key intelligence into a single structured representation
2. THE Compressed_Summary SHALL contain: entity reference, summary text, key facts extracted, confidence level, source memory count, and generation timestamp
3. WHEN a Compressed_Summary exists for an entity, THE Context_Builder SHALL prefer the Compressed_Summary over individual raw memories unless the query specifically requires granular detail
4. WHEN new memories are added to an entity that has an existing Compressed_Summary, THE Memory_Compression_Engine SHALL mark the summary as stale
5. WHEN a stale Compressed_Summary is detected during the Reflection Engine cycle, THE Memory_Compression_Engine SHALL regenerate the summary incorporating new memories
6. THE Memory_Compression_Engine SHALL preserve links to source memories so that drill-down from summary to raw memories remains possible
7. WHEN contexta.explain(memoryId) is called for a Compressed_Summary, THE system SHALL return the list of source memories that contributed to the summary
8. THE Memory_Compression_Engine SHALL reduce context token usage by a minimum of 60% compared to delivering all source memories individually

### Requirement 25: Semantic Cluster Engine

**User Story:** As a developer, I want related memories to be automatically grouped into semantic clusters, so that retrieval can pull coherent groups of related intelligence rather than isolated memories.

#### Acceptance Criteria

1. THE Semantic_Cluster_Engine SHALL automatically identify groups of semantically related memories and entities based on embedding proximity and graph relationships
2. THE Semantic_Cluster_Engine SHALL assign a descriptive name to each Semantic_Cluster (e.g., "Backend Infrastructure", "Authentication System", "Revenue Goals")
3. WHEN a retrieval query matches a Semantic_Cluster, THE Retrieval_Engine SHALL offer the full cluster as a retrieval unit in addition to individual memories
4. THE Semantic_Cluster_Engine SHALL require a minimum of 3 related memories or entities to form a cluster
5. WHEN new memories are stored that are semantically related to an existing cluster, THE Semantic_Cluster_Engine SHALL add the memory to the cluster
6. WHEN a memory in a cluster is invalidated or archived, THE Semantic_Cluster_Engine SHALL remove the memory from the cluster and re-evaluate cluster validity
7. THE Semantic_Cluster_Engine SHALL update clusters during Reflection Engine cycles and on-demand when new high-importance memories are stored
8. THE Semantic_Cluster_Engine SHALL scope clusters to the user_id and organization_id of the constituent memories

### Requirement 26: Dream Cycle Engine

**User Story:** As a developer, I want contexta to autonomously evaluate and improve its own retrieval quality by generating synthetic questions and identifying knowledge gaps, so that the memory system continuously self-improves.

#### Acceptance Criteria

1. THE Dream_Cycle_Engine SHALL execute as a periodic background worker (configurable interval, default weekly)
2. WHEN the Dream_Cycle_Engine runs, THE Dream_Cycle_Engine SHALL generate synthetic questions based on known entities and their relationships (e.g., "What database does user use for project X?")
3. WHEN a synthetic question is generated, THE Dream_Cycle_Engine SHALL attempt to retrieve an answer using the Retrieval_Engine
4. WHEN a retrieval attempt returns a confident answer, THE Dream_Cycle_Engine SHALL evaluate the answer for accuracy, completeness, and consistency with known facts
5. WHEN a retrieval attempt fails or returns low-confidence results, THE Dream_Cycle_Engine SHALL generate a Missing_Memory_Candidate identifying the knowledge gap
6. WHEN a Missing_Memory_Candidate is generated, THE Dream_Cycle_Engine SHALL store the candidate for surfacing to the agent during future interactions (e.g., prompting "What deployment provider does project X use?")
7. WHEN a retrieval succeeds during dream evaluation, THE Dream_Cycle_Engine SHALL increment the utility score of memories that contributed to the answer
8. WHEN a retrieval fails during dream evaluation, THE Dream_Cycle_Engine SHALL decrement the utility score of memories that were retrieved but did not contribute
9. THE Dream_Cycle_Engine SHALL generate compressed summaries for entities where retrieval quality is high but context delivery is inefficient
10. THE Dream_Cycle_Engine SHALL log all evaluation results (questions generated, retrieval outcomes, gap identifications) in the audit trail

### Requirement 27: Context Planner

**User Story:** As a developer, I want contexta to intelligently allocate token budget across memory categories when building context, so that agents receive the most valuable intelligence within their context window limits.

#### Acceptance Criteria

1. THE Context_Planner SHALL accept a total token budget parameter specifying the maximum context size for the requesting agent
2. THE Context_Planner SHALL allocate the token budget across memory categories using configurable default weights: projects (35%), goals (20%), preferences (10%), episodic events (15%), facts (15%), relationships (5%)
3. WHEN a developer provides custom budget allocation weights, THE Context_Planner SHALL use the custom weights instead of defaults
4. THE Context_Planner SHALL fill each category allocation with the highest-relevance memories that fit within the category token budget
5. WHEN a category has fewer relevant memories than its allocated budget allows, THE Context_Planner SHALL redistribute unused tokens to other categories proportionally
6. THE Context_Planner SHALL prefer Compressed_Summaries over raw memories when both are available and the summary fits within the budget allocation
7. WHEN Semantic_Clusters are relevant to the query, THE Context_Planner SHALL allocate cluster context as a unit rather than splitting cluster members across categories
8. THE Context_Planner SHALL include the User_Model and relevant Project_Snapshots within the allocated budget before filling with individual memories
9. THE Context_Planner SHALL report the actual token usage per category in the context response metadata for developer visibility
